# 设计原则

redis追求快速读写，不需要保证数据强一致

# 集群模式

集群模式主要分成一下三种，架构逐渐变复杂

1. 主从：配置简单，主写从读
2. 哨兵：避免主节点故障无法自动恢复
3. cluster：数据分片

| 模式 | 特点             | 优点                       | 缺点                                     |
| ---- | ---------------- | -------------------------- | ---------------------------------------- |
| 主从 | 主写从读         | 简单<br>提升读性能         | 不支持故障转移<br>单点写瓶颈<br>不能分片 |
| 哨兵 | 主从基础上加哨兵 | 主节点自动切换             | 单点写瓶颈<br>不能分片                   |
| 集群 | 横向可扩容       | 可以分片<br>多个主节点可写 | 复杂<br>数据不是强一致的                 |

## 哨兵

哨兵`sentinel`是单独的进程

### 主节点切换流程

1. 主观下线：`sentinel`周期`PING`主节点，主节点不返回，认为下线
2. 客观下线：多数`sentinel`认为主节点下线
3. 哨兵选举：`sentinel`进行选主
4. `Redis`选主：主`sentinel`选择从节点升级【主节点偏移量小；在线】
5. 从节点升级主节点；其余节点都改为从

## 集群【cluster】

### 主节点切换流程

1. 多个主通信
2. 某个主疑似下线【主观下线】
3. 投票【客观下线】
4. 从节点开始选举
5. 主节点投票
6. 晋升为新主

### 分片写

通过`hash`映射槽位，突破单点的内存上限



# 扩展

`redis`这几种集群模式的区别和优缺点都很简单，但是做成集群，就必然会遇到选主问题，`redis`这里实现的简单。但是还有很多开源框架也有类似的功能需求，比如 `zk`、`kafka`、`etcd` 他们都是如何实现的

# Raft【强共识算法】

## 目的

分布式一致性算法：多个机器在可能宕机、网络抖动、消息乱序情况下，对状态达成一致性

灵魂：多数确认才是提交成功

## 模块

1. leader选取
2. 日志复制
3. 安全

## 角色与任期【raft时间轴】

### 角色

1. leader：唯一的写入库
2. candidate：竞选者
3. follower：跟随者

### 任期

目的：解决旧主复活、避免 历史幽灵

核心操作

1. 每次选取任期就会加一
2. 永远以更大的term为准
3. 节点看到更大的term，会退化成follower

## leader选取

1. follower等待 leader心跳【随机超时】
   1. 自己进化成 candidate
   2. term加一
   3. 给自己投票，向其余节点发起投票请求
2. 从节点投票：规则
   1. term大于自己当前的
   2. 自己在此term还没投过票
   3. 候选人日志不能比自己的旧

## 日志复制

1. 主节点写入日志
2. 并行发给从节点：当前term、新的日志、上一次的日志【preindex】
   1. 从节点检查：preindex有数据；term不比自己的小，开始追加写
   2. 如果preindex没数据：leader需要找到上次对齐的位置，开始让从同步
3. 多数子节点commit，主节点才可以commit

## 哪些技术用到了类似的思想

|                | 是否raft            | 是否强一致   | 是否写入多数确认 |
| -------------- | ------------------- | ------------ | ---------------- |
| zk             | 否【ZAB，类似技术】 | 是           | 是               |
| etcd           | 是                  | 是           | 是               |
| kafka【KRaft】 | 部分                | 原数据强一致 | 原数据时         |
| redis sentical | 否                  | 否           | 否               |
| redis cluster  | 否                  | 否           | 否               |
