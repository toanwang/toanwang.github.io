# IO模型

`IO`事件模型

所有的网络服务入口都是`IO`事件模型，本质都是要解决以下问题

1. 如何知道数据到了
2. 怎么处理数据
3. 如何管理状态

|          | web-netty                    | web-tomcat | redis                  | mysql                  |
| -------- | ---------------------------- | ---------- | ---------------------- | ---------------------- |
| 处理策略 | 多 Reactor分片               | 线程池阻塞 | 单线程Reactor          | 线程池                 |
| 瓶颈     |                              |            | 数据在内存，单线程足够 | 磁盘是瓶颈，必须多线程 |
| 特点     | 少线程<br>事件驱动<br>高并发 |            | 简单<br>IO和执行同线程 |                        |

## Reactor逻辑

`redis`

```
while (true):
	epoll_wait (等待IO事件)
	处理socket读写
	处理定时任务
```

`netty`

```
bossGroup (处理链接，1个或者少量线程)
workerGroup (处理实际IO)
	eventloop1:【可以理解为一个简单的redis事件模型】
	eventloop2:
	eventloopn:
		线程 (eventloop下面单线程处理)
		selector
		channelList (一个channel只能绑定一个eventloop)
		taskQueue
		scheculeTaskQueue
```

* `redis`是全局串行
* `netty`是分片串行

```
              bossGroup (通常 1 个线程)
                      │
               接受新连接
                      ↓
        ┌───────── workerGroup ─────────┐
        │        │        │        │
    worker1  worker2  worker3  worker4
```

# 消息队列模型

现在接触到的模型基本分两种，`pull`和`push`，区别在于谁控制消费节奏

* `pull`：消费者主动拉取【`kafka`、`redis blockqueue`】
* `push`：服务器主动推送【`redis subpub 模式`】

`push` 实时性会稍微高一点

`pull`比较稳定，不会有激增流量

实际遇到的各种各样的问题，都和消费模型有关系，因为`kafka`本身十分稳定

### reblance

什么情况下会触发`reblance`

1. `consumer`下线【心跳参数】
   1. 服务异常重启之后被立即拉起，再重新`reblance`
2. `consumer`卡主【`poll`参数】
   1. `poll`模式是阻塞的，业务流程耗时过长，导致`kafka`认为消费者卡主，会触发`reblance`
   2. 如果是长时间的业务操作，建议用工作线程池区异步处理

## xxljob

### 简单总结

1. 启动时通过注解扫描，维护`jobName->jobBean`
2. 服务启动将执行器注册到调度器上
3. 调度器根据策略选择一个执行器：数据库 + 时间轮
   1. 预读未来五秒的任务；根据触发器，计算每秒的任务，放入时间轮
   2. 每秒触发任务列表；计算下一次的`trigger_next_time`
4. 执行器执行的线程模型如下
   1. `jobThreadRepository：jobId->JobThread`
   2. 每个`jobId`对应一个`jobThread`：包括 处理器、队列、状态等

### 常见问题

#### 多实例如何保证串行

1. 调度器是本身保证的，只需要考虑执行器
2. 调度策略必须选择`hash`，因为`xxljob`只能感知到当前执行器的任务状态，无法感知到其他节点的任务状态
   1. 阻塞策略只对当前的执行器生效

#### 长任务会有什么问题

1. 频繁调度+长时间任务，导致任务阻塞队列变大
2. 体感：定时任务一直不执行，与预期不符

和`kafka`类似，解决此类问题

1. 调度只负责触发，实际任务用业务线程池去处理

# Raft 强共识算法

## 目的

分布式一致性算法：多个机器在可能宕机、网络抖动、消息乱序情况下，对状态达成一致性

灵魂：多数确认才是提交成功

## 模块

1. leader选取
2. 日志复制
3. 安全

## 角色与任期【raft时间轴】

### 角色

1. leader：唯一的写入库
2. candidate：竞选者
3. follower：跟随者

### 任期

目的：解决旧主复活、避免 历史幽灵

核心操作

1. 每次选取任期就会加一
2. 永远以更大的term为准
3. 节点看到更大的term，会退化成follower

## leader选取

1. follower等待 leader心跳【随机超时】
   1. 自己进化成 candidate
   2. term加一
   3. 给自己投票，向其余节点发起投票请求
2. 从节点投票：规则
   1. term大于自己当前的
   2. 自己在此term还没投过票
   3. 候选人日志不能比自己的旧

## 日志复制

1. 主节点写入日志
2. 并行发给从节点：当前term、新的日志、上一次的日志【preindex】
   1. 从节点检查：preindex有数据；term不比自己的小，开始追加写
   2. 如果preindex没数据：leader需要找到上次对齐的位置，开始让从同步
3. 多数子节点commit，主节点才可以commit
